## What was built

Five integrated modules + a single webhook receiver:

```

├── api/webhook.py                  # Shopify + Woo receiver (HMAC + idempotency + BackgroundTasks)

│   ├── events.py                   # normalisation of Shopify + Woo payloads

│   ├── llm.py                      # Euri gpt-4o-mini wrapper with prompt rendering
│   ├── run_logger.py               # runs/YYYY-MM-DD-<module>.jsonl writer

│   └── senders/
```

## Disambiguations decided without blocking

The ATLAS-PROMPT's rules-of-engagement listed four open questions. I resolved each with the most conservative, clearly-documented choice rather than stop the build — reasoning logged below so Angelina can reverse any of them in minutes:

| Question                                          | Decision                                                         | Rationale |
|---------------------------------------------------|------------------------------------------------------------------|-----------|
| Shopify vs WooCommerce vs both                    | Both, Shopify primary. `/webhook/shopify` + `/webhook/woocommerce` routes. | ATLAS-PROMPT mentions "Shopify/Woo" throughout. Event normaliser abstracts both into one shape. |
| XLwear vs generic template                        | Generic template. XLwear data can be dropped into `.env` + `knowledge/`. | README sells this as a Rs.50K–2L D2C package. XLwear flagged as *possible* cross-use, not required. |
| Review platforms                                  | Shopify JSON + local fixtures in MVP. Google / Trustpilot are stubs that raise `NotImplementedError` with a clear message. | Avoids wiring paid-API integrations without Angelina's go-ahead; stub contract makes adding them a 1-file change. |
| Inventory auto-order                              | **Alerts-only.** `suggest_reorder.suggest()` returns `auto_order_enabled: false`. Flag is explicit so it can be flipped later with a written green-light. | Dhruv's build-contract rule #5 (never auto-charge) + spirit of it — auto-ordering is auto-spending. |

If any of these are wrong, flag it and I'll re-dispatch.

## Lessons (candidates for learning-hub/ERRORS.md)

- **Script-invocation path setup:** modules that import sibling project code (e.g. `from tools.x import y`) break when run as `python modules/x/y.py` unless the project root is on `sys.path`. Pattern adopted: a 5-line walk-up prelude in every CLI file that locates the project root by looking for `tools/_bootstrap.py`. (Not logged as a full ERRORS.md entry — this is pattern-level, captured in the code.)
- **Discount-recovery invariant as code, not just docs:** the repo contains zero imports that could place an order or create a charge. Invariants live in types/absence of code, not in comments.
